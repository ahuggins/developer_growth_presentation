Day 1 to Code God

How to grow as a Developer

Created by Andy Huggins

In my own career as a coder/programmer/developer I remember starting out, and some things being really difficult. I would see people online, follow screencasts and wonder "These people had to start learning similar to what I learned starting out...how is it they seem to know so much more?"

That question never really got answered, but as I approach my 10th year doing code, I find that I somehow just know stuff. Or maybe not that I know it, but I seem to just be able to run with something new much faster than I used to.

So the rest of this talk is sort of my theory as to why that is, which I hope might provide some pointers for others.

Also, Code God is kind of a joke, I think most people have to put in time to learn things, and there are a few, very rare, who just pick up these details early and fast.

Secret Sauce

What separates experienced devs from beginners?

Deep Knowledge

Looking back, I know that I have a deeper understanding of how code works. And that comes with time, but also comes with wrestling with code in your mind. Copy and Pasting from Stack Overflow is ok, but I think to really get better, you need to understand why that code solves the problem. Wrestling with why it solves the problem, is how you gain deeper understanding of code in general.

This is the first tip, while looking for the cause of an error, don't just copy/paste and see if it fixes the error. Take a little time to think about what the copied code is doing, why it is doing it. It's not to say you can never copy/paste code, but to gain that experience, you need to understand the fix.

The key is to not just read them in a weekend and be done, try taking what you learn and apply that to your code. Maybe you go back and just refactor some code, or the next time you are writing code, you think about this practices.

Beyond books, follow tutorials/screencasts, but focus on what the code is doing, not just recreating what is onscreen, or copying it.

Debugging

Debugging is a crucial skill to top programmers/devs. At some point errors will happen, knowing what is happening and how to fix it is difficult. But when errors happen, before googling, try to think about the changes you made that are now erroring. Why would it do that? Again, I am emphasising wrestling with the code to understand it.

Some error messages are cryptic, and you will need to Google them. I picked up a tip a while ago, a tutorial I was following the dev was narrating what he was expecting. Sometimes he expected an error and was surprised if he didn't get it, and actually found a bug because there was no error. This indicates a good mental model of the code and how the code should behave. Try stating what you expect to happen before running your code, and see if it behaves properly. This is a good way to learn.

Build Toy Apps

  • Another way to gain deeper knowledge, is to build toy apps. A toy app, is something you kind of know will never be used in production, you're going into it with the goal of learning. The great thing about them is that they are a fresh starting point. There is no overhead/legacy code you have to deal with. There are no deadlines. You get to make all the decisions and you are free to build what you think is important.
  • You can practice making decisions, and you don't have to learn how a big existing system works in order to make changes.
  • You can practice new techniques and see how those play out in terms of working within a system.
  • Another great thing is that at the beginning, this toy app can have a small focus. Which hopefully means there is little code, and it is easy to reason about. This makes working within easier.
  • This is also a good place to learn about testing, since there should not be a "who knows where to start" situation.

Write Tests

One of the things I am working on adding to my toolbelt is testing. Testing is a good practice because to test code you write, you have to think about how it works and what you need to assert to verify it works as expected.

This can give you perspective about your code, and how it works. That perspective is useful when you have to write new code.

Break Habits

"Everything looks like a nail, when all you have is a hammer"

When you start out, you usually learn the basics:

  1. Maybe you learn how to Hello World.
  2. Variables
  3. Control Structures (IF, while, etc)
  4. Data Structures (Array, struct, collection, etc)

And often, this is a imperative/procedural language. But to some degree, these are considered bad practices as you advance. So I think you have to break these habits in order to improve. Think of If, If If, Else, If, Foreach, If, Else, Else.....This is considered bad code, but it's one of the first things you learn basically.

So learn them, because they have their place, but be aware that they may not be the best approach.

Mentality

Confidence

Staying Power

This is a difficult one, To some degree I think what separates a new dev from an experienced dev is a bit of mentality. Experienced devs know they can figure things out, new devs struggle with knowing enough. Imposter syndrome is a big issue, because people don't want to look dumb. But working on a system that has been built over years, and you are just walking in...the fact that you may not know something is really an indication of poor documentation than your ability.

Read Source Code

Another great way to learn is to read existing source code. Github is an amazing resource where you can access code written by all kinds of programmers. You should seek out people who have a reputation for good code, and then go read theirs.

What you can gain here, is you can see how people structure their code, or maybe a trick...keep in mind not all code is great, but in reading many people's code, you can develop a sense of what is good/bad/acceptable and this affects the way you code things.
